NOUS Academy uses a checked-exception strategy: every custom exception class extends `Exception`, which means the compiler enforces that callers either catch or propagate each one. Domain objects (e.g., `Reserva`, `Membresia`, `Grupo`) throw exceptions when business rules are violated. `AcademiaController` propagates these exceptions up to the UI layer, which is the only place they should be caught and displayed to the user.

## Exception handling pattern

```java
// UI layer (view / menu class)
try {
    controller.crearEstudiante(nombre, apellidos, telefono, correo,
                               tipoDocumento, numeroDocumento, usuario, contrasena);
} catch (ElementosVacios e) {
    System.out.println("All fields are required.");
} catch (CorreoNoValido e) {
    System.out.println("Email must end with nousacademy.co");
} catch (Duplicados e) {
    System.out.println("A student with that document or username already exists.");
}
```

<Note>
  Never catch exceptions inside a domain class or the controller unless the recovery is a domain concern (e.g., archiving a membership before replacing it). Let all other exceptions propagate to the UI.
</Note>

<AccordionGroup>

<Accordion title="User and validation exceptions">

| Class | Default message | When thrown |
|---|---|---|
| `ElementosVacios` | `Campos vacios no permitidos` | A required string field is empty, or a required object reference is `null` |
| `ElementoNoExistente` | *(varies)* | A lookup by document ID or code finds no matching entity in the controller's arrays |
| `Duplicados` | *(varies)* | A user with the same document number or username already exists when creating a new account; or a student is added to a group they already belong to |
| `NombreNoValido` | *(varies)* | `nombre` or `apellidos` is shorter than 2 characters |
| `CorreoNoValido` | *(varies)* | Email does not contain `@` or does not end with the required domain |
| `ValoresNegativos` | *(varies)* | A numeric field (phone, document number, hours, salary, capital) is negative |
| `LongitudNoValida` | *(varies)* | `telefono` or `numeroDocumento` has fewer than 5 characters |
| `FechaInvalida` | *(varies)* | A date is `null` or falls before today (for memberships, contracts) or before the current moment (for reservation start times) |

</Accordion>

<Accordion title="Membership exceptions">

| `TipoMembresiaInvalida` | *(varies)* | A `null` type or `PERSONALIZADA` is used with the defined-plan constructor |
| `EstudianteSinMembresia` | *(varies)* | An operation requires an active membership but the student's `membresia` field is `null` |

</Accordion>

<Accordion title="Contract exceptions">

|---|---|---|

| `DineroInsuficienteParaContratar` | `No hay suficiente capital para contratar` | The academy's `capital` is less than `salarioBase * meses` when `asignarContrato()` is called |

## Exception class structure

All exceptions follow the same minimal pattern:

```java
public class ElementosVacios extends Exception {
    public ElementosVacios() {
        super("Campos vacios no permitidos");
    }
}
```

They carry no additional fields. Error details are encoded entirely in the exception type itself, and the UI layer selects the appropriate user-facing message based on which class is caught.
